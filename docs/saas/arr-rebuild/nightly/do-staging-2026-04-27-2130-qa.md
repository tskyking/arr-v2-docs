## Findings

### 1. Upload and ingestion are working on DO staging

The live API accepted the sanitized workbook and produced consistent import, summary, ARR, movement, customer, cube, review, and CSV export responses.

### 2. Dashboard/API correspondence is internally consistent

The dashboard values observed in browser correspond to live API outputs for the same import.

### 3. Main UX/data-semantics mismatch: “Latest ARR” means final imported period, not current-date ARR

The dashboard labels the final period in the import's term range as “Latest ARR.” That is technically consistent with the API, but it can be misleading during review because a workbook with contracts extending into future periods may show a future runoff month instead of today/current operating ARR.

In this QA workbook:

- A user may expect the dashboard headline to highlight around `2026-04` or `2026-07`, where workbook rows visibly add up to `$180.5K`–`$204.6K` ARR.
- The dashboard instead highlights `2027-06` `$24.1K` because that is the last recognized period in the range.
- The customer roster's `currentArr` has the same behavior: it reports the latest snapshot period, so `QA Beta Biosciences` shows `currentArr=0` even though it contributes `$60.2K` ARR in `2026-04` through `2027-03`.

Recommendation: either rename this concept to “Latest imported period ARR” / “Ending ARR,” or add an “As of month” selector/default that uses current month or most recent invoice month for headline/customer current ARR.

### 4. One-year end-date convention inflates annual ARR slightly

The engine computes ARR as `amount / (end - start days) * daysInYear`. For common subscription terms entered as `2026-01-01` → `2026-12-31`, this uses 364 days, so `$120,000` becomes `$120,329.67`. For `2026-04-01` → `2027-03-31`, `$60,000` becomes `$60,164.84`.

That is mathematically consistent with current code, but business users often interpret end dates as inclusive. This small inflation could contribute to “numbers do not match spreadsheet” concerns.

Recommendation: decide whether subscription end date is inclusive, exclusive, or normalized to anniversary date. If inclusive, recognition should probably use `(end - start + 1 day)` for day count and active-month logic should be tested accordingly.

### 5. Hash routing/deep-link issue on staging

The root app works and hash-route dashboards work. Direct path dashboards return 404. This may be acceptable if hash routing is intentional, but QA/deploy notes should use hash URLs or the static deploy should be configured with SPA fallback for path routing.

## Open questions / follow-ups

- Should dashboard headline/current customer ARR default to current calendar month, latest invoice month, selected month, or final imported period?
- Should ARR day-count treatment consider subscription end dates inclusive?
- Should staging support direct path SPA fallback, or should all documentation and UI links use `/#/...` URLs only?
