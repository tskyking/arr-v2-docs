## Executive summary

Reviewed the overnight DO staging reports, especially the 02:30 Coordinator handoff. The highest-priority finding was P0 staging persistence: generated import links disappeared between the 01:30 QA upload and 02:30 Coordinator check.

Because ARR-V2 is currently file-backed and DigitalOcean App Platform runtime files should not be treated as durable demo persistence, this pass did **not** pretend to solve persistence with another local `DATA_DIR` path. Instead it made the risk explicit and smoke-testable, improved the missing-import dashboard UX, and tightened the draft DO static-site fallback config.

## Implementation notes

### 1. Backend storage diagnostics for staging smoke tests

Added file-store diagnostics so QA/Coordinator can distinguish “API is healthy” from “dashboard links are durable.”

- `GET /api/health` now includes a `storage` object.
- `GET /api/health/storage` returns storage diagnostics directly.
- Tenant-aware `GET /api/tenants/:tenantId/health/storage` returns diagnostics for a tenant.

Diagnostic fields:

```json
{
  "kind": "file",
  "dataDirConfigured": true,
  "writable": true,
  "importCount": 0,
  "durability": "ephemeral-risk",
  "warning": "Import persistence is file-backed on a local/runtime filesystem. Use managed durable storage before relying on shared dashboard links."
}
```

The implementation writes/deletes a tiny probe file and counts persisted import JSON files. It intentionally does not expose filesystem paths.

### 2. Missing-import dashboard UX

When a dashboard summary request fails, the dashboard now shows an explicit “Import unavailable” recovery panel instead of only `Summary error: Import not found`.

The panel explains that staging uploads are file-backed and may disappear after deploy/restart/cold start until durable storage is configured, and gives two concrete actions:

- re-upload workbook
- open the sample dashboard

It also reminds users to share hash-route links (`/#/dashboard/...`) until direct route fallback is applied.

### 3. Draft DigitalOcean deploy config/docs

Updated `apps/arr-v2/deploy/digitalocean-app.yaml` and README to remove ambiguity around persistence:

- Added comments that `DATA_DIR=/workspace/data` is prototype-only and not durable enough for shared links.
- Documented the need for Managed DB or Spaces-backed persistence before demos depend on reopen-after-restart behavior.
- Added `catchall_document: index.html` to the draft static-site spec so direct dashboard/review/cube paths can fall back to the React shell when the spec is applied.

- **Current DO live app needs autodeploy/redeploy before `/api/health/storage` exists on staging.** Pre-deploy live `/api/health` still returns only `status` and `ts`.
- **Draft `digitalocean-app.yaml` still has placeholder repo fields.** It should not be applied as-is.
